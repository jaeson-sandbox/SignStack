## 1. Project Context Analysis

### Application Type

SignStack is a **client-side-only single-page application** hosted as a static bundle. The Next.js server is used only for `next dev` (local development) and static build generation. There is no runtime server, no API routes, and no server-side rendering of PDF content.

### Privacy Constraint (Non-Negotiable)

> The architecture structurally prevents Document content from leaving the user's device.

This means: No network requests carry PDF bytes. No API routes receive file data. No third-party SDK that uploads data. Any architecture decision that would require transmitting Document content — for any reason, including performance — is a hard block.

### Functional Requirements Summary

| FR Group | Count | Key Constraint |
|----------|-------|----------------|
| PDF Upload (FR-1 to FR-3) | 3 | File picker + drag-drop; 25 MB hard limit; file validation before render |
| PDF Rendering (FR-4, FR-5) | 2 | All pages, vertically scrollable; first page ≤ 3s for ≤ 20 page / ≤ 10 MB doc |
| Signature Creation (FR-6 to FR-10) | 5 | Draw (canvas) + Typed (Clean/Script/Formal); one session Signature |
| Signature Overlay (FR-11 to FR-15) | 5 | Drag, resize, delete, multiple overlays across pages |
| PDF Export (FR-16 to FR-19) | 4 | Client-side embedding; ≤ 5s for 20 pages + 5 overlays |
| Privacy & Disclaimer (FR-20, FR-21) | 2 | Zero bytes transmitted; persistent UI disclaimer |

### Non-Functional Requirements Summary

- **NFR-P1/P2/P3:** Render ≤ 3s, export ≤ 5s, initial load ≤ 5s
- **NFR-B1:** Chrome 110+, Firefox 110+, Safari 16+ desktop
- **NFR-PV1/PV2:** Zero Document bytes to any server; no analytics accessing Document content
- **NFR-A1/A2/A3:** WCAG 2.1 AA on all controls outside canvas; typed path fully keyboard-accessible; overlay keyboard nudge
- **NFR-R1/R2:** Offline after initial load; export errors surfaced gracefully

### Session Data Lifecycle (Architecture Contract)

All of the following exist **exclusively in React component state** (memory) and nowhere else:
- `ArrayBuffer` of original PDF bytes
- Rendered page canvases (generated by pdfjs-dist)
- Signature PNG data URL (drawn strokes or typed text rendered to canvas)
- Overlay position/size objects

**Nothing is written to `localStorage`, `IndexedDB`, cookies, or any server.** Page refresh, tab close, or new Document load clears all state. This is both the privacy guarantee and the implementation contract.

## 3. Core Architectural Decisions

### AD-1: Next.js as Application Shell Only

**Decision:** Next.js is used purely as a development server and static build tool. The application is a client-side SPA. No API routes, no server actions, no server-side rendering of Document content.

**Consequences:**
- All PDF-processing components carry the `'use client'` directive
- `pdfjs-dist` and `pdf-lib` are **never imported in server context** — use dynamic imports with `ssr: false` for all PDF components
- The app exports as a static bundle (`next build` produces a static-exportable output)
- `next.config.ts` must configure `output: 'export'` if deploying statically, or omit for `next dev` local use

**Pattern for PDF components:**
```typescript
// Any file that imports pdfjs-dist or pdf-lib must be client-only
'use client';
import dynamic from 'next/dynamic';

// For the top-level PDF rendering component:
const PDFScrollArea = dynamic(
  () => import('@/components/editor/PDFScrollArea'),
  { ssr: false }
);
```

### AD-2: PDF Rendering via react-pdf v10

**Decision:** Use `react-pdf`'s `<Document>` and `<Page>` components for rendering. Do not implement a custom pdfjs-dist rendering pipeline.

**pdfjs Worker Setup (v5):** pdfjs-dist v5 requires the worker to be configured. The exact approach for react-pdf v10 + pdfjs v5 must be verified in the library README. The worker file path has changed in v5. A common pattern:

```typescript
// src/lib/pdf/pdfWorker.ts — loaded once at app startup
import { pdfjs } from 'react-pdf';
pdfjs.GlobalWorkerOptions.workerSrc = new URL(
  'pdfjs-dist/build/pdf.worker.min.mjs',
  import.meta.url
).toString();
```

**Verify this path against the installed pdfjs-dist v5 package before implementing.**

**Page rendering width:** Pages render to a fixed container width (`containerWidth`, approximately 794px — the max-width of the PDF scroll area). The `<Page width={containerWidth}>` prop controls render size.

### AD-3: PDF Coordinate Mapping (Critical)

**Decision:** The screen-to-PDF coordinate transformation is centralized in a single utility (`src/lib/pdf/coordinateMapper.ts`). No component or export function computes coordinates inline.

**Coordinate systems:**
- **Screen:** pixels, origin top-left of rendered page, Y increases downward
- **PDF (pdf-lib):** points (1 point = 1/72 inch), origin bottom-left of page, Y increases upward

**Scale factor computation:**
```
scaleFactor = renderedPageWidthPx / pdfPageWidthPt
```

**Coordinate transformation (screen → PDF):**
```
pdfX = overlay.screenX / scaleFactor
pdfY = pdfPageHeightPt - (overlay.screenY / scaleFactor) - (overlay.screenHeight / scaleFactor)
pdfWidth = overlay.screenWidth / scaleFactor
pdfHeight = overlay.screenHeight / scaleFactor
```

**Per-page rendered dimensions** must be stored when each page finishes rendering. Use react-pdf's `onRenderSuccess` callback (or equivalent in v10 — verify API):

```typescript
const [pageDimensions, setPageDimensions] = useState<Map<number, { width: number; height: number }>>(new Map());
// Store rendered px dimensions keyed by pageIndex (0-based)
```

**PDF intrinsic dimensions** (in points) are accessed via pdfjs-dist's `PDFPageProxy.getViewport({ scale: 1 })` which returns `{ width, height }` in points. This requires direct pdfjs-dist access alongside react-pdf rendering.

### AD-4: Signature Embedding Pipeline

**Decision:** Both Drawn and Typed Signatures are converted to PNG data URLs and embedded via `pdf-lib` as raster images. The embedding pipeline is:

```
Drawn Signature:
  react-signature-canvas.toDataURL('image/png') → PNG data URL

Typed Signature:
  Render text on an offscreen <canvas> in the selected Google Font → canvas.toDataURL('image/png') → PNG data URL

Export:
  1. Load original PDF bytes via pdf-lib: PDFDocument.load(originalArrayBuffer)
  2. For each Overlay (sorted by pageIndex):
     a. Get page: pdfDoc.getPage(overlay.pageIndex)
     b. Embed PNG: pdfDoc.embedPng(overlay.signatureDataUrl) → PDFImage
     c. Map coordinates: coordinateMapper.screenToPdf(overlay, pageDimensions, pdfIntrinsicDimensions)
     d. Draw image: page.drawImage(pdfImage, { x, y, width, height })
  3. Serialize: const bytes = await pdfDoc.save()
  4. Trigger download: create Blob → URL.createObjectURL → anchor click → revoke URL
```

**Output filename:** `{original-name}-signed.pdf` (OQ-6 resolved by UX spec).

### AD-5: Session State Management via React Context + useReducer

**Decision:** Application state is managed via a single React Context with a `useReducer` hook. No external state management library is used. State shape is typed and centralized.

**Rationale:** The pre-selected tech stack does not include a state manager. React Context + useReducer is sufficient for a single-session, single-document, client-only app. Avoids adding a dependency not approved in the stack.

**State shape:**

```typescript
// src/types/index.ts

export interface DocumentState {
  file: File | null;
  arrayBuffer: ArrayBuffer | null;
  pageCount: number;
  // Rendered page pixel dimensions (0-based page index → {widthPx, heightPx})
  pageDimensionsPx: Map<number, { width: number; height: number }>;
  // PDF intrinsic point dimensions (0-based page index → {widthPt, heightPt})
  pageIntrinsicPt: Map<number, { width: number; height: number }>;
}

export interface SignatureState {
  dataUrl: string | null; // PNG data URL; null = no signature created yet
  type: 'drawn' | 'typed' | null;
}

export interface Overlay {
  id: string;          // crypto.randomUUID()
  pageIndex: number;   // 0-based
  x: number;           // px from page left
  y: number;           // px from page top
  width: number;       // px
  height: number;      // px
}

export interface AppState {
  document: DocumentState;
  signature: SignatureState;
  overlays: Overlay[];
  selectedOverlayId: string | null;
  ui: {
    isSignatureModalOpen: boolean;
    isExporting: boolean;
    uploadError: string | null;
    exportError: string | null;
  };
}
```

**Actions:**

```typescript
// src/store/appActions.ts
export type AppAction =
  | { type: 'DOCUMENT_LOADED'; payload: { file: File; arrayBuffer: ArrayBuffer; pageCount: number } }
  | { type: 'DOCUMENT_CLEARED' }
```

### AD-6: Overlay Keyboard Navigation (NFR-A3 Resolution)

**Decision:** Keyboard nudge is implemented via a document-level `keydown` event listener attached when `selectedOverlayId` is non-null and the modal is closed.

**Behavior:**
- `ArrowUp/Down/Left/Right`: nudge selected Overlay 1px in the direction
- `Shift+Arrow`: nudge 8px
- `Delete` or `Backspace`: delete selected Overlay
- All nudges are clamped to page bounds
- Listener is removed when `selectedOverlayId` is null or modal opens (to prevent interference with modal keyboard controls)

**Implementation location:** `src/hooks/useKeyboardOverlay.ts`. This hook is consumed by `PDFScrollArea.tsx`.

```typescript
// src/hooks/useKeyboardOverlay.ts
export function useKeyboardOverlay(
  selectedOverlayId: string | null,
  isModalOpen: boolean,
  dispatch: React.Dispatch<AppAction>
): void {
  useEffect(() => {
    if (!selectedOverlayId || isModalOpen) return;
    const handleKeyDown = (e: KeyboardEvent) => { /* ... */ };
    window.addEventListener('keydown', handleKeyDown);
    return () => window.removeEventListener('keydown', handleKeyDown);
  }, [selectedOverlayId, isModalOpen, dispatch]);
}
```

### AD-7: Google Font Deferred Loading (Typed Signature)

**Decision:** The three signature Google Fonts (Caveat, Dancing Script, Pinyon Script) are loaded on-demand when the user first activates the Type tab in the Signature modal. They are not in the initial bundle.

**Implementation:** Dynamically insert a `<link>` element into `document.head` on first Type tab activation. Track load state to show a spinner while fonts load.

```typescript
// src/lib/signature/fontLoader.ts
const FONTS_HREF = 'https://fonts.googleapis.com/css2?family=Caveat&family=Dancing+Script&family=Pinyon+Script&display=swap';

let fontsLoaded = false;
let fontsLoading: Promise<void> | null = null;

export async function loadSignatureFonts(): Promise<void> {
  if (fontsLoaded) return;
  if (fontsLoading) return fontsLoading;
  fontsLoading = new Promise((resolve) => {
    const link = document.createElement('link');
    link.rel = 'stylesheet';
    link.href = FONTS_HREF;
    link.onload = () => { fontsLoaded = true; resolve(); };
    link.onerror = () => resolve(); // Fail silently; preview will use fallback
    document.head.appendChild(link);
  });
  return fontsLoading;
}
```

**Typed Signature → PNG:** After fonts load, render the user's text on an offscreen `<canvas>` using the selected font, then call `canvas.toDataURL('image/png')` to produce the PNG data URL.

```typescript
// src/lib/signature/typedSignatureRenderer.ts
export function renderTypedSignatureToPng(
  text: string,
  fontFamily: string, // e.g., "'Caveat', cursive"
  fontSize: number    // e.g., 64 (px; larger = higher quality PNG)
): string {
  const canvas = document.createElement('canvas');
  const ctx = canvas.getContext('2d')!;
  ctx.font = `${fontSize}px ${fontFamily}`;
  const metrics = ctx.measureText(text);
  canvas.width = Math.ceil(metrics.width) + 16;
  canvas.height = fontSize * 1.4;
  ctx.font = `${fontSize}px ${fontFamily}`; // Re-set after resize
  ctx.fillStyle = '#1E1B4B'; // --color-ink
  ctx.fillText(text, 8, fontSize);
  return canvas.toDataURL('image/png');
}
```

### AD-8: File Size Limit — OQ-3 Resolution

**Decision:** Keep the 25 MB hard limit. Rationale and implementation:

**Memory analysis for 25 MB PDF:**
- Original ArrayBuffer: 25 MB
- pdfjs-dist parsing overhead: ~2–5× PDF size in memory (50–125 MB)
- Rendered canvases: use lazy rendering (only render pages in viewport ± 1 page buffer) to cap canvas memory
- pdf-lib export: copies original bytes + small signature image data
- **Estimated peak memory: 100–200 MB** — acceptable for modern desktop browsers (≥8 GB RAM)

**Implementation requirements:**
- Validate file size before loading into memory (read `file.size` before `file.arrayBuffer()`)
- Implement **lazy page rendering** in `PDFScrollArea`: use `IntersectionObserver` to render pages only when they approach the viewport. Unmount canvases that are more than 2 pages away from the current viewport position.
- Display a clear error message for files over 25 MB (FR-2)

**File validation order (FR-2):**
1. Check `file.size > 25 * 1024 * 1024` → reject with size error
2. Check MIME type: `file.type !== 'application/pdf'` → reject with type error
3. Read first 4 bytes → check for `%PDF` header → reject corrupt/non-PDF
4. Pass to pdfjs-dist loading (catches encrypted PDFs, malformed structure)

### AD-9: Touch / Stylus Support — OQ-2 Resolution

**Decision:** Touch and stylus input on the Signature Canvas is supported at the library level (`react-signature-canvas` handles touch events). No additional implementation is required. Touch input is **not explicitly QA-tested** for MVP 1. Mobile layout is not optimized.

**Architectural implication:** No touch-specific code. No viewport meta tag changes beyond standard responsiveness. No touch event handlers in Overlay components (react-rnd handles touch drag natively — verify in v10.5.3 changelog).

### AD-10: No API Routes

**Decision:** The `src/app/api/` directory does not exist in this project. Any file placed there would execute on the server and must not receive Document content. To prevent accidental server-side imports, PDF processing utilities (`src/lib/pdf/`) must include the `'use client'` directive at the module level or be imported only by `'use client'` components.

### AD-11: Overlay Default Placement

**Decision:** After Signature confirmation, the new Overlay is placed on the **page currently most visible in the viewport** (determined by `IntersectionObserver` or scroll position). Default position: bottom-right quadrant, inset 5% from each edge. Default size: 200px wide × proportional height based on signature aspect ratio (minimum 40px high).

**Implementation:** `PDFScrollArea` tracks which page is most visible and passes `currentVisiblePageIndex` to the Overlay placement logic.

## 5. Implementation Patterns & Consistency Rules

### 5.1 Naming Conventions

| Element | Convention | Example |
|---------|-----------|---------|
| Component files | PascalCase `.tsx` | `SignatureModal.tsx` |
| Hook files | camelCase with `use` prefix `.ts` | `useOverlays.ts` |
| Utility/lib files | camelCase `.ts` | `coordinateMapper.ts` |
| TypeScript interfaces | PascalCase | `interface Overlay` |
| TypeScript types | PascalCase | `type AppAction` |
| CSS custom properties | kebab-case | `--color-accent` |
| Action type strings | SCREAMING_SNAKE_CASE | `'OVERLAY_MOVED'` |
| Page indexes | Always 0-based in code | `pageIndex: 0` |
| IDs | `crypto.randomUUID()` — never sequential integers | `'3f2504e0-...'` |

### 5.2 File Header Requirement

Every `.tsx` component file that uses browser-only APIs must begin with:

```typescript
'use client';
```

Every file in `src/lib/pdf/` and `src/lib/signature/` must either:
- Begin with `'use client'`, OR
- Be imported exclusively from `'use client'` components

**Never import `pdfjs-dist` or `pdf-lib` in a file without `'use client'`.** Vitest unit tests are exempt (they run in jsdom, not Next.js).

### 5.3 State Update Patterns

- **All state mutations go through `dispatch`** — no direct state modification
- **Overlays are immutable** — moves and resizes produce new Overlay objects (`OVERLAY_MOVED` creates a replacement, not a mutation)
- **IDs are stable** — an Overlay's `id` never changes after `OVERLAY_ADDED`
- **Never derive IDs from array index** — overlays can be deleted and re-ordered

### 5.4 Error Handling Patterns

**User-facing errors:** All errors shown to the user are strings stored in `AppState.ui.uploadError` or `AppState.ui.exportError`. Components read these from context and render inline error banners. No `alert()`, no `console.error()` for user-visible errors.

**Internal errors (unexpected):** Wrap pdf-lib and pdfjs-dist operations in try/catch. Dispatch `EXPORT_ERROR` or `UPLOAD_ERROR` with a user-readable message. Log the original error to `console.error` for debugging.

**Error message format (user-facing):**
- Specific: "This file is not a PDF. Please select a PDF file." (not "Invalid file")
- Actionable: "The file exceeds the 25 MB limit. Try a smaller PDF."
- Non-technical: Avoid library names, stack traces, or error codes in user messages

### 5.5 Loading State Patterns

| State | Where | What triggers it |
|-------|-------|-----------------|
| PDF loading | `AppState.ui` (add `isDocumentLoading: boolean`) | After file selected, before `DOCUMENT_LOADED` |
| Font loading | Local state in `TypeTab` | `loadSignatureFonts()` promise pending |
| Exporting | `AppState.ui.isExporting` | Between `EXPORT_START` and `EXPORT_SUCCESS/ERROR` |

Loading states disable the triggering control and show a spinner or skeleton. No global loading overlay in MVP 1.

### 5.6 Coordinate Units Rule

**Rule:** Variables holding pixel values end with `Px`. Variables holding PDF points end with `Pt`. Variables holding generic "overlay" units (used before conversion) use no suffix.

```typescript
const widthPx: number;   // screen pixels
const widthPt: number;   // PDF points
const { x, y, width, height } = overlay;  // pre-conversion, in screen px
```

This prevents the most common embedding bug: mixing up screen pixels and PDF points.

### 5.7 Component Props Patterns

- Props types are defined as named `interface` in the same file (for small components) or in `src/types/index.ts` (for shared types)
- No `any` types. Use `unknown` with type guards if the type is truly dynamic.
- Event handlers passed as props are named `on{Event}`: `onClose`, `onConfirm`, `onDelete`
- No component renders another component's internal state — all shared state goes through context

### 5.8 Test Placement

Co-located tests are forbidden. All tests live under `tests/`:

```
tests/
├── lib/
│   ├── pdfValidator.test.ts
│   ├── coordinateMapper.test.ts
│   └── typedSignatureRenderer.test.ts
└── components/
    └── UploadZone.test.tsx
```

Test files end with `.test.ts` or `.test.tsx`. Integration tests (requiring a real browser environment) are out of scope for MVP 1 — Vitest with jsdom handles unit and component tests.

### Integration Boundaries

**Internal data flow:**

```
User action → Component → Custom Hook → dispatch(Action)
                                           ↓
                                        appReducer → new AppState
                                           ↓
                                        Context consumers re-render
```

**No component calls `pdf-lib` or `pdfjs-dist` directly.** Library calls are encapsulated in `src/lib/` modules, called from `src/hooks/`.

**Component → Hook boundaries:**
- `UploadZone` → `usePdfDocument`
- `EditorToolbar` → `useExport` (for download), dispatch `SIGNATURE_MODAL_OPEN`
- `DrawTab` → `useSignature` (capture PNG from canvas)
- `TypeTab` → `useSignature` + `fontLoader`
- `SignatureOverlay` → dispatch `OVERLAY_MOVED`, `OVERLAY_RESIZED`, `OVERLAY_DELETED`, `OVERLAY_SELECTED`
- `PDFScrollArea` → `useKeyboardOverlay`, renders `PDFPageRenderer` + `SignatureOverlay` per page

## 7. Development Workflow (Formal)

This section is a required part of the architecture. All AI agents implementing SignStack stories must follow this workflow exactly. It is also documented in `CLAUDE.md`.

### Per-Story Workflow

For every implementation story, execute this sequence:

1. **Pre-flight:** Run `git status`. If working tree is dirty (uncommitted changes), stop and report — do not proceed until the state is explained and resolved.

2. **Sync:** `git checkout main && git pull` — always branch from an up-to-date main.

3. **Branch:** `git checkout -b story/descriptive-story-name` — use kebab-case, prefix with `story/`.

4. **Implement:** Write only what the current story requires. Do not touch adjacent code, add speculative abstractions, or refactor unrelated areas.

5. **Verify:**
   - `npm run lint` — must pass with zero errors
   - `npm run build` — must succeed
   - `npm test` (if tests exist) — must pass

   Fix failures before creating the PR. Do not create a PR with known failures.

6. **Commit:** `git add <specific files> && git commit -m "feat: <story description>"` — stage specific files by name, not `git add .`.

7. **Push:** `git push -u origin story/descriptive-story-name`

8. **PR:** `gh pr create --title "feat: ..." --body "..."` with body containing:
   - Summary of what was implemented
   - Story ID / AC reference
   - Manual test steps
   - Checks run (lint, build, test)
   - Screenshots if UI changed
   - Risks or follow-up items

### Hard Rules

- **Never push directly to `main`** — all implementation stories go through a PR
- **Never merge a PR** unless the user explicitly says "merge"
- **Never use `--no-verify`** — hooks must pass
- **Stop if `gh` is not authenticated** — report what needs to be fixed before proceeding
- **Stop if git is dirty** before starting a new story — do not overwrite in-progress work

### Branch Naming

| Story type | Branch pattern |
|-----------|---------------|
| Implementation story | `story/upload-zone-component` |
| Bug fix | `fix/overlay-bounds-clamp` |
| Infrastructure | `infra/ci-workflow-setup` |

## 8. Architecture Validation

### Coherence Validation

**Decision compatibility:**
- All decisions are consistent with the privacy constraint (AD-1, AD-10 enforce no server contact)
- pdf-lib (PDF writing) and pdfjs-dist/react-pdf (PDF reading/rendering) are complementary, non-conflicting
- React Context + useReducer is compatible with Next.js App Router client components
- react-rnd and react-signature-canvas are client-side libraries with no server-side dependencies
- Tailwind v4 CSS configuration is compatible with Next.js 16 PostCSS pipeline

**Pattern consistency:**
- Coordinate unit naming convention (AD-3, §5.6) prevents the most common embedding bug
- All state flows through dispatch (§5.3) prevents inconsistent state mutation patterns
- `'use client'` rules (§5.2, AD-1) prevent SSR/client mismatch errors

**Structure alignment:**
- Every FR maps to a specific file (§6.3) — no FR is architecturally unaddressed
- Library calls are encapsulated in `src/lib/` (§6.4 integration boundaries) — testable in isolation

### Requirements Coverage

**All 21 FRs are covered:**
- FR-1 to FR-21: each maps to a specific component/module in §6.3
- No FR requires server-side code

**All NFRs are addressed:**

| NFR | Architectural Response |
|-----|----------------------|
| NFR-P1 (first page ≤ 3s) | Lazy rendering via IntersectionObserver (AD-8) |
